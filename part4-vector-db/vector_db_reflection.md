# Part 4 — Vector Databases

## Vector DB Use Case

Traditional keyword-based search would be fundamentally insufficient for a law firm's plain-English contract search system. Consider a concrete example: a lawyer searches for "termination clauses" in a 500-page contract. A keyword search finds only sections containing the word "termination." It completely misses sections titled "Exit Provisions," "Contract Dissolution Terms," or "Early Discontinuation Rights" — all describing the same concept but using different words. In a legal context, missing even one such section could have serious professional consequences.

This is precisely the problem **vector databases** solve. A vector database converts text into dense numerical embeddings — high-dimensional vectors encoding semantic meaning — using a model such as `sentence-transformers/all-MiniLM-L6-v2`. In this vector space, "termination clauses" and "exit provisions" are geometrically close because the model has learned they represent the same legal concept. When a lawyer types a plain-English question, it is converted to a query vector and the database returns the most semantically similar contract sections using approximate nearest-neighbour search — regardless of exact wording. This is impossible with keyword search.

The practical architecture works as follows: the 500-page contract is split into overlapping text chunks, each embedded and stored in a vector database such as **Pinecone**, **Weaviate**, or **pgvector**. At query time, the top-k most relevant chunks are passed to a language model to generate a cited, synthesised answer — a pattern called **Retrieval-Augmented Generation (RAG)**. This makes plain-English contract search both semantically accurate and legally traceable, which no keyword system can achieve.
